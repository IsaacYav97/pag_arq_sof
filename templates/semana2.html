Semana 2: Documentación Arquitectónica (ADRs)

"Code tells you HOW. Comments tell you WHAT. Documentation tells you WHY."
La documentación no es burocracia, es infraestructura de conocimiento compartido. Evita que el equipo futuro se pregunte: "¿Por qué demonios hicieron esto así?".

📚 La Filosofía de la Documentación

🎯 ¿Qué captura un ADR?

Un Architecture Decision Record captura el problema (Contexto), la solución elegida (Decisión) y los trade-offs aceptados (Consecuencias). Evita la pérdida de memoria institucional.

👥 ¿Para quién es?

  • Devs: Para saber cómo implementar.
  • QA: Para saber qué probar.
  • Nuevos: Para el Onboarding rápido.

📝 Anatomía de un ADR Perfecto

1. Contexto

Las fuerzas en juego. ¿Qué problema técnico o de negocio intentamos resolver? ¿Qué restricciones tenemos (tiempo, dinero, escalabilidad)?

2. Decisión

La tecnología o patrón elegido de forma clara y accionable. Justificada con razones técnicas frente a las alternativas descartadas.

3. Consecuencias

Lo bueno y lo malo. ¿Qué deuda técnica asumimos? ¿Qué mitigaciones planeamos? (Recuerda la regla del Trade-off).

🛠️ Laboratorio Interactivo: Redactando tu primer ADR

Basado en el ejercicio práctico de la clase. Lee el escenario y piensa qué pondrías en cada sección antes de revelar la respuesta del arquitecto.

Escenario del Proyecto: Tu equipo está construyendo el nuevo catálogo de productos del Yavirac. Necesitan una base de datos. Los productos tienen estructuras muy diferentes (unos tienen talla, otros tienen peso, otros tienen voltaje) y el catálogo cambia constantemente. Tienen que decidir entre MySQL (Relacional) y MongoDB (NoSQL).
Ver ADR Propuesto: Elección de Base de Datos para Catálogo

[ADR-001] Uso de MongoDB para el Catálogo de Productos

Estado: Aceptado | Fecha: Hoy

1. Contexto

El catálogo de productos requiere almacenar ítems con atributos altamente variables. Un esquema rígido en SQL nos obligaría a hacer migraciones constantes de tablas o usar patrones complejos (Entity-Attribute-Value) que penalizan el rendimiento de lectura.

2. Decisión

Hemos decidido utilizar MongoDB (Base de datos NoSQL orientada a documentos). Descartamos MySQL porque la flexibilidad del esquema JSON nos permite insertar nuevos tipos de productos sin alterar la estructura de la base de datos.

3. Consecuencias
  • Positivas: Alta flexibilidad (Escalabilidad de negocio). Tiempo de desarrollo más rápido para nuevos productos.
  • Negativas (Trade-offs): Perdemos las transacciones ACID fuertes (Fiabilidad) y las uniones complejas (JOINs) nativas de SQL. Los desarrolladores deberán garantizar la integridad de los datos a nivel de aplicación.
