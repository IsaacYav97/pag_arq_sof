Semana 5: Arquitectura Reactiva y Evolutiva

Estilos III: Sistemas que reaccionan en tiempo real y están diseñados para el cambio constante.

⚡ Arquitectura Orientada a Eventos (EDA)

Sistemas que reaccionan a cambios de estado en tiempo real en lugar de depender de peticiones directas.

El Modelo Pub/Sub

  • Productores: Generan el evento.
  • Canales: Brokers intermediarios como Kafka o RabbitMQ.
  • Consumidores: Reaccionan al evento.

Beneficios y Desafíos

  • Ventaja Máximo desacoplamiento. Los servicios no necesitan conocerse entre sí.
  • Reto La trazabilidad se vuelve compleja. Requiere patrones como Saga para mantener la consistencia en la transaccionalidad distribuida.

⚖️ Patrón CQRS

Command Query Responsibility Segregation propone usar modelos distintos para actualizar información y para consultarla.

Separación de Responsabilidades

  • Commands Tareas específicas de negocio que alteran el estado (Ej: "Reservar Habitación").
  • Queries Consultas optimizadas que devuelven DTOs sin lógica de dominio.

¿Por qué implementarlo?

Las lecturas suelen ser 100x más frecuentes que las escrituras. Al separar responsabilidades, logras un escalado independiente, modelos de lectura desnormalizados para máxima velocidad y un control estricto de seguridad.

🧬 Arquitectura Evolutiva

El software no es estático. Se enfoca en la "Evolutividad" como una característica arquitectónica de primer nivel, evitando la rigidez del diseño inicial.

Fitness Functions (Funciones de Aptitud)

Son los guardianes del diseño. Métricas objetivas que evalúan qué tan cerca está la arquitectura de sus objetivos:

$$f(a) \rightarrow [0, 1]$$

Se integran en el pipeline de CI/CD para detectar desviaciones en rendimiento, seguridad o deuda técnica automáticamente.

💻 Laboratorio: Event-Bus en Python

Simularemos un sistema donde una orden de compra dispara eventos a Inventario y Notificaciones sin llamadas directas.

event_bus.py 
class EventBus:
        def __init__(self):
            self.subscribers = {}

        def subscribe(self, event_type, callback):
            if event_type not in self.subscribers:
                self.subscribers[event_type] = []
            self.subscribers[event_type].append(callback)

        def publish(self, event_type, data):
            if event_type in self.subscribers:
                for callback in self.subscribers[event_type]:
                    callback(data)

    # --- Uso del Event Bus ---
    bus = EventBus()

    def notificar_inventario(data):
        print(f"📦 Inventario actualizado para: {data['item']}")

    bus.subscribe("orden_creada", notificar_inventario)

    # Emitiendo el evento asíncrono
    bus.publish("orden_creada", {"item": "Laptop", "qty": 1})
> 📦 Inventario actualizado para: Laptop
